#### 1. LLVM整体框架与执行流程

LLVM可以理解为“前端 + 优化器 + 后端”的模块化编译基础设施。

- 前端（如 Clang）：把源码转成 LLVM IR。
- 中端（Optimizer）：在 LLVM IR 上做大量 Pass 优化（标量优化、循环优化、内联等）。
- 后端（CodeGen）：把 LLVM IR 降到目标机器指令（x86/ARM/RISCV），再汇编链接生成可执行文件。

典型流程：

1. 词法/语法/语义分析 -> AST。
2. AST Lowering -> LLVM IR（SSA 形式）。
3. IR Pass Pipeline 优化（-O1/-O2/-O3 对应不同优化强度）。
4. 指令选择、寄存器分配、指令调度。
5. 生成目标文件并链接。

面试可说要点：LLVM核心价值是“统一 IR + 可插拔 Pass + 多目标后端复用”。

#### 2. TVM整体框架与执行流程

TVM是面向 AI 编译的系统，核心目标是把模型在不同硬件上高效落地。

常见组件视角：

- 前端导入：PyTorch/ONNX/TensorFlow 模型导入 Relay/Relax IR。
- 图级优化：算子融合、常量折叠、布局变换、消除冗余算子。
- 算子级调度：TIR（Tensor IR）描述循环与访存，结合 AutoTVM/MetaSchedule 自动搜索最优 schedule。
- 后端代码生成：生成 CUDA/LLVM/C 等目标代码并运行。

典型流程：

![图片](https://tvm.hyper.ai/assets/images/02-how-to_03-optimize-large-language-model_1-4bf7ea93309ace0834c5ffd5ae89f2cd.png)

1. 模型导入并转成高层图 IR。
   1. 表示计算图的高层 Relax 函数
   2. 用于描述张量程序的低层 TensorIR 函数
2. 执行可组合优化
   1. 图优化（融合、裁剪、量化相关变换）。
   2. 张量程序优化
   3. 对关键算子做 schedule 优化（tiling、vectorize、unroll、tensorize）。
3. （构建并进行通用部署）Lower 到低层 IR，生成目标代码并部署。

面试可说要点：TVM相较通用编译器更强调“图优化 + 算子调度 + 硬件感知自动调优”。

#### 3. 具体的图优化（Graph-level Optimization）

图优化关注算子之间的全局数据流，目标是减少访存、减少 kernel 启动、降低延迟。

高频图优化：

- 算子融合：Conv + BN + ReLU 融合，减少中间张量落地。
- 常量折叠：编译期可算的子图提前计算。
- 公共子表达式消除：重复子图只算一次。
- 死代码消除：输出无关分支删除。
- 布局变换与传播：NCHW/NHWC 调整以匹配硬件友好访存。

简化示例：

优化前：

- y1 = Conv(x, w)
- y2 = BatchNorm(y1, gamma, beta)
- y3 = ReLU(y2)

优化后：

- y3 = FusedConvBNReLU(x, w, gamma, beta)

收益：减少中间内存读写和 kernel launch 次数。

#### 4. 具体的算子优化（Operator-level Optimization）

算子优化关注单个算子的实现细节，目标是提高吞吐、降低访存瓶颈。

高频手段：

- Loop Tiling/Blocking：提升 cache 命中率。
- 向量化（SIMD）：一次处理多元素。
- 并行化：多线程或多核分块计算。
- 循环展开（Unroll）：减少分支和循环控制开销。
- 访存优化：共享内存/预取/对齐访问。
- Winograd、im2col+GEMM 等卷积变体。

伪代码示例（矩阵乘分块）：

```text
for ii in 0..N step B:
	for jj in 0..N step B:
		for kk in 0..N step B:
			// 在小块上做乘加，提升局部性
			C[ii:ii+B][jj:jj+B] += A[ii:ii+B][kk:kk+B] * B[kk:kk+B][jj:jj+B]
```

面试可说要点：图优化决定“算什么和怎么拼”，算子优化决定“每个算子如何跑得快”。

#### 5. IR层次及其作用

IR分层的本质是把复杂问题拆阶段处理，不同层次保留不同信息，便于逐层优化与降级。

- 高层 IR：保留语义信息（张量形状、算子语义、控制流结构），便于做全局和语义相关优化。
- 中层 IR：更接近通用编译优化模型（SSA、CFG），便于做数据流分析、常量传播、DCE 等。
- 低层 IR：接近硬件执行（循环、内存地址、线程映射），便于做寄存器、指令和访存优化。

核心作用：

1. 解耦前端语言与后端硬件。
2. 每层做最适合自己的优化。
3. 提升编译器可维护性和可扩展性。

#### 6. 常见IR层次与区别

可按“抽象程度”理解：

- AST（语法树）：最接近源码语法结构，语义分析友好，但不适合直接做底层性能优化。
- HIR/Graph IR（如 Relay/MLIR 高层方言）：表达算子图和高层语义，适合融合、布局变换、自动微分。
- MIR/SSA IR（如 LLVM IR）：控制流与数据流清晰，适合通用编译优化。
- LIR/TIR（低层循环与访存 IR）：显式循环和内存访问，适合硬件相关优化。
- Machine IR / ISA：目标机指令级表示，最终生成汇编/机器码。

对比维度：

- 抽象级别：AST/HIR 高，LIR/Machine IR 低。
- 硬件相关性：高层弱，低层强。
- 优化类型：高层偏语义和图优化，低层偏访存、并行和指令优化。

一句话总结：从 AST 到 Machine IR 是“语义逐步减少、执行细节逐步增加”的过程。

#### 1. 智能指针

智能指针是 RAII 的典型应用，用对象生命周期管理堆内存，避免手写 `new/delete`。

- `std::unique_ptr`：独占所有权，不能拷贝，只能移动，开销最小。
- `std::shared_ptr`：共享所有权，内部引用计数，最后一个引用释放时自动析构。
- `std::weak_ptr`：不增加引用计数，配合 `shared_ptr` 打破循环引用。

```cpp
#include <iostream>
#include <memory>
using namespace std;

struct Node {
	int v;
	Node(int x) : v(x) { cout << "Node(" << v << ") ctor\n"; }
	~Node() { cout << "Node(" << v << ") dtor\n"; }
};

int main() {
	unique_ptr<Node> p1 = make_unique<Node>(1);
	// unique_ptr<Node> p2 = p1; // 错误：不能拷贝
	unique_ptr<Node> p2 = move(p1); // 所有权转移

	shared_ptr<Node> s1 = make_shared<Node>(2);
	shared_ptr<Node> s2 = s1; // 引用计数 +1
	cout << s1.use_count() << "\n"; // 2
}
```

面试可补一句：工程里优先 `make_unique` / `make_shared`，尽量不直接裸 `new`。

#### 2. 运算符重载

运算符重载是给自定义类型定义运算语义，本质是函数调用，不能改变运算符优先级、结合性和操作数个数。

常见写法：

- 成员函数：`operator+=`、`operator[]` 等。
- 非成员函数：`operator+`、`operator<<` 等（通常更对称）。

```cpp
#include <iostream>
using namespace std;

class Vec2 {
public:
	int x, y;
	Vec2(int x, int y) : x(x), y(y) {}

	Vec2& operator+=(const Vec2& rhs) {
		x += rhs.x;
		y += rhs.y;
		return *this;
	}
};

Vec2 operator+(Vec2 lhs, const Vec2& rhs) {
	lhs += rhs;
	return lhs;
}

ostream& operator<<(ostream& os, const Vec2& v) {
	return os << "(" << v.x << ", " << v.y << ")";
}

int main() {
	Vec2 a(1, 2), b(3, 4);
	cout << (a + b) << "\n"; // (4, 6)
}
```

#### 3. 什么是内存泄露？如何解决？

内存泄露：申请的内存不再可达（没有指针能访问），但未释放，导致进程内存持续增长。

常见原因：

- `new` 后忘记 `delete`。
- 异常路径提前返回，释放逻辑没执行。
- `shared_ptr` 循环引用。

解决方式：

- 优先 RAII（智能指针、容器对象）替代裸指针。
- 明确资源所有权（谁创建谁释放，或转移所有权）。
- 使用工具排查：ASan、Valgrind、LSan。

```cpp
void bad() {
	int* p = new int[100];
	// 忘记 delete[] p; -> 泄露
}

void good() {
	auto p = std::make_unique<int[]>(100);
	// 离开作用域自动释放
}
```

#### 4. 指针指向对象数组，释放时如何释放？

如果是 `new[]` 分配的对象数组，必须用 `delete[]` 释放，不能用 `delete`。

```cpp
struct A {
	A()  { std::cout << "A ctor\n"; }
	~A() { std::cout << "A dtor\n"; }
};

int main() {
	A* arr = new A[3];
	delete[] arr; // 正确：会调用 3 次析构

	// delete arr; // 错误：未定义行为
}
```

补充：如果是 `malloc` 分配，必须 `free`；`new/delete` 与 `malloc/free` 不能混用。

#### 5. 构造函数？拷贝构造函数是什么？

- 构造函数：对象创建时初始化成员。
- 拷贝构造函数：用“同类型已存在对象”初始化新对象，典型签名：`T(const T&)`。

触发拷贝构造的常见场景：

- `T b = a;`
- 函数按值传参
- 函数按值返回（可能被拷贝省略优化）

```cpp
#include <cstring>

class Buffer {
	char* data;
public:
	Buffer(const char* s = "") {
		data = new char[strlen(s) + 1];
		strcpy(data, s);
	}

	// 拷贝构造：深拷贝，避免多个对象共享同一块堆内存
	Buffer(const Buffer& other) {
		data = new char[strlen(other.data) + 1];
		strcpy(data, other.data);
	}

	~Buffer() { delete[] data; }
};
```

面试可扩展：

- 如果类管理资源（堆内存、文件句柄、锁等），通常需要遵循“三法则/五法则”（拷贝构造、拷贝赋值、析构；再加移动构造、移动赋值）。
- 现代 C++ 推荐优先用 `std::string`、`std::vector`、智能指针管理资源，减少手写拷贝逻辑。
